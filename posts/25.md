WebGPU는 **웹과 네이티브(Dawn, wgpu)** 에서 모던 GPU에 직접 매핑되도록 설계된 API다. Vulkan처럼 모든 것을 수동으로 제어하지는 않지만, Bind Group·Render Bundle·Indirect Draw 같은 최적화 포인트는 명확하다.

> **용어 정리**
> - **Bind Group**: 셰이더에 바인딩되는 리소스 묶음 (Vulkan의 descriptor set에 대응)
> - **Render Bundle**: 미리 기록해 둔 반복 커맨드 시퀀스
> - **WGSL**: WebGPU Shading Language

---

## 1. Bind Group 설계

바인딩 비용을 줄이려면 **변경 빈도가 낮은 리소스를 낮은 group 인덱스**에 둔다.

```cmdstack
group 0 ← 프레임별 (카메라, 환경)
group 1 ← 머티리얼별
group 2 ← 오브젝트별 (모델 행렬)
```

### PipelineLayout 재사용

여러 파이프라인이 같은 `GPUPipelineLayout`을 공유하면, 공통 slot은 재바인딩할 필요가 없다.

예: 파이프라인 X(A,B,C)와 Y(A,D,C) → **slot 1만** 다시 바인딩.

### Bind Group 재사용 + Dynamic Offsets

동일 리소스 조합은 `createBindGroup`을 한 번만 호출한다. 오브젝트별 데이터는 **dynamic offset**으로 구분한다.

```javascript
// Offset only — no re-bind
pass.setBindGroup(0, frameBindGroup);
pass.setBindGroup(2, objectBindGroup, [dynamicOffset]);
```

사용 가능한 dynamic uniform/storage buffer 개수는 device limit을 확인해야 한다.

## 2. Render Bundle

`GPURenderBundle`은 **변하지 않는 지오메트리**의 draw 시퀀스를 미리 인코딩해 매 프레임 재실행한다.

```javascript
// Encode once
const enc = device.createRenderBundleEncoder(descriptor);
enc.setPipeline(pipeline);
enc.setBindGroup(0, bindGroup);
enc.setVertexBuffer(0, vertexBuffer);
enc.draw(vertexCount);
const bundle = enc.finish();

// Reuse every frame
renderPass.executeBundles([bundle]);
```

**주의**:
- 실행 후 파이프라인·버텍스 버퍼 등 바인딩 상태는 **초기화**된다
- `depthReadOnly` / `stencilReadOnly` 모드 지원
- 정적 메시·UI·타일맵 등 반복 draw에 효과적

Vulkan의 Secondary Command Buffer + 고정 draw 조합과 비슷한 역할이다.

## 3. 파이프라인

### 비동기 생성

```javascript
const pipeline = await device.createRenderPipelineAsync(descriptor);
```

메인 스레드 블로킹을 피한다. 로딩 화면·백그라운드 워커에서 미리 컴파일하는 패턴이 일반적이다.

### Feature / Limit 요청

필요한 기능만 요청한다. 불필요한 feature는 어댑터 선택을 제한할 수 있다.

```javascript
const device = await adapter.requestDevice({
  requiredFeatures: ['texture-compression-bc', 'depth-clip-control'],
  requiredLimits: { maxBindGroups: 8 },
});
```

브라우저는 동일 셰이더+상태 조합에 대해 **내부 파이프라인 캐시**를 유지한다.

---

## 4. 텍스처 압축

런타임 압축은 **불가**하다. 빌드 타임에 플랫폼별 포맷으로 준비해야 한다.

| Feature | 포맷 | 플랫폼 |
|---------|------|--------|
| `texture-compression-bc` | BC1~BC7 | 데스크톱 |
| `texture-compression-etc2` | ETC2, EAC | 모바일 (GLES 3.0+) |
| `texture-compression-astc` | ASTC 4×4~12×12 | 모바일·일부 데스크톱 |

`texture-compression-bc-sliced-3d`, `texture-compression-astc-sliced-3d`로 3D 텍스처 압축도 지원한다. 어댑터 feature를 확인한 뒤 에셋 파이프라인에 반영한다.

## 5. Compute Shader (WGSL)

### Workgroup 크기

고정된 정답은 없다. `maxComputeInvocationsPerWorkgroup`, 공유 메모리 사용량, 레지스터 압력, occupancy를 같이 보고 튜닝한다.

```wgsl
@compute @workgroup_size(8, 8, 1)
fn main(@builtin(global_invocation_id) gid: vec3<u32>) {
    if (gid.x >= width || gid.y >= height) { return; }
    // ...
}
```

### Shared Memory

`var<workgroup>`으로 워크그룹 내 공유. `maxComputeWorkgroupStorageSize`(기본 16384바이트) 제한을 확인한다.

### 버퍼 선택

| 용도 | 바인딩 |
|------|--------|
| 읽기 전용 | `uniform` 또는 `read-only-storage` |
| 읽기/쓰기 | `storage` (`read_write`는 느릴 수 있음) |

### Indirect Dispatch

```javascript
computePass.dispatchWorkgroupsIndirect(indirectBuffer, offset);
```

GPU가 워크그룹 수를 결정한다. 컬링·LOD 결과를 indirect buffer에 쓴 뒤 dispatch하는 패턴과 잘 맞는다.

## 6. Indirect Drawing

```javascript
renderPass.drawIndirect(indirectBuffer, offset);
renderPass.drawIndexedIndirect(indirectBuffer, offset);
```

- `indirect-first-instance` feature로 non-zero `firstInstance` 지원
- Compute로 프러스텀 컬링 → indirect buffer 기록 → draw

Vulkan의 `vkCmdDrawIndirect`와 동일한 GPU-driven 렌더링 흐름이다.

```cmdstack
Compute Pass ← visible count · draw args 기록
---
Render Pass ← drawIndirect(indirectBuffer)
---
CPU draw loop 없음
```

## 7. Depth / Stencil

| 기능 | 설명 |
|------|------|
| **depth-clip-control** | clip space Z 범위 제어, 원근 정확도 |
| **Read-only depth/stencil** | Render Bundle에서 깊이 버퍼 재사용 시 동기화 감소 |
| **depth24plus** | 24비트 (일부 구현은 float32) |
| **depth32float** | float 깊이 (24비트의 상위집합은 아님) |
| **depth16unorm** | 낮은 정밀도, 빠른 렌더링 |

Depth Pre-pass 후 메인 패스에서 `depthReadOnly`로 읽으면 불필요한 depth 쓰기를 막을 수 있다.

## 8. Occlusion Query · 프로파일링

```javascript
renderPass.beginOcclusionQuery(queryIndex);
// ... draws ...
renderPass.endOcclusionQuery();
```

가시성 결과로 **다음 프레임 컬링**에 활용한다. `timestamp-query` feature가 있으면 `timestampWrites`로 GPU 구간 시간도 측정할 수 있다.

---

## 9. Vulkan과 비교 (요약)

| 영역 | Vulkan | WebGPU |
|------|--------|--------|
| Draw 감소 | Indirect + DGC + Secondary CB | Indirect + Render Bundle |
| 바인딩 | Descriptor Buffer, Push Constants | Bind Group 계층, Dynamic Offsets |
| 파이프라인 | Pipeline Cache, Shader Objects | 비동기 생성, 브라우저 캐시 |
| VRS / Mesh Shading | 확장으로 지원 | 표준 미지원 |
| 텍스처 압축 | 전 포맷 | feature flag로 선택 |
| 동기화 | 명시적 배리어 | usage scope 기반 암시적 |

WebGPU는 **안전성과 이식성**을 우선하고, Vulkan은 **최대 제어와 확장**을 우선한다. 웹에서는 Render Bundle + Bind Group 설계 + Indirect Draw가 핵심 레버다.

---

## 10. 최적화 체크리스트

- Bind Group 계층 설계 (변경 빈도순)
- 동일 PipelineLayout 재사용
- Render Bundle로 반복 커맨드 최적화
- 텍스처 압축 feature 감지·활용
- Workgroup Size 하드웨어 맞춤
- Indirect Draw / Dispatch
- `createRenderPipelineAsync`로 블로킹 방지
- Dynamic Offsets로 재바인딩 최소화
- Depth Pre-pass 또는 Read-only Depth
