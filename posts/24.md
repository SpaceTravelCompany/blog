3D 렌더링 성능은 결국 **CPU가 얼마나 일을 덜 하느냐**, **GPU가 하드웨어를 얼마나 잘 쓰느냐**, **메모리 대역폭을 얼마나 아끼느냐** 세 축으로 갈린다. Vulkan이든 WebGPU든 API가 달라도, 이 세 가지는 공통이다.

이 글은 API에 독립적인 **3D 그래픽스 최적화 전략**을 정리한다.

> **용어 정리**
> - **Draw Call**: CPU가 GPU에 "이걸 그려라"고 명령을 내리는 단위
> - **Overdraw**: 같은 픽셀에 여러 번 그려지는 현상 (낭비)
> - **LOD (Level of Detail)**: 거리에 따라 메시·텍스처 복잡도를 낮추는 기법

---

## 1. CPU 측 최적화 — Draw Call 줄이기

CPU 병목은 보통 **상태 변경**과 **드로우 제출 횟수**에서 온다. GPU가 놀고 CPU만 바쁜 전형적인 패턴이다.

| 기법 | 설명 |
|------|------|
| **Draw Call Batching** | 동일 파이프라인·바인딩 상태의 오브젝트를 묶어 한 번에 제출 |
| **Instancing** | 같은 메시를 변환 행렬만 바꿔 반복 렌더링 |
| **Indirect Drawing** | draw 파라미터를 GPU 버퍼에 두고 GPU가 직접 읽음 |
| **GPU-Driven Rendering** | 프러스텀·오클루전 컬링, LOD 선택을 GPU에서 수행 |
| **Command Buffer 재사용** | 매 프레임 바뀌지 않는 커맨드를 재기록하지 않음 |

```cmdstack
전통적 렌더링 ← CPU: 컬링 → 상태 설정 → draw × N
---
배칭/인스턴싱 ← CPU: 상태 1회 → draw 1회 (N개 오브젝트)
---
GPU-Driven ← CPU: indirect buffer 제출 1회 · GPU: 컬링+draw 결정
```

**실전 팁**: 오브젝트를 **파이프라인 → 머티리얼 → 메시** 순으로 정렬하면 상태 전환 횟수가 줄어든다.

---

## 2. GPU 측 최적화 — 그릴 것만 그리기

GPU 시간은 "얼마나 많이 그리느냐"보다 **불필요한 픽셀·삼각형을 얼마나 빨리 버리느냐**가 더 중요하다.

| 기법 | 설명 |
|------|------|
| **Frustum Culling** | 카메라 시야 밖 오브젝트 제외 |
| **Occlusion Culling** | 다른 오브젝트에 가려진 것 제외 |
| **Backface Culling** | 카메라를 등진 면 제외 |
| **LOD** | 거리에 따라 메시·셰이더 복잡도 조절 |
| **Mesh Shading** | Task/Mesh Shader로 GPU에서 메시 단위 처리 |
| **Meshlet 렌더링** | 메시를 작은 클러스터로 나눠 단위 컬링·LOD |

```cmdstack
카메라 시야
---
Frustum Culling ← 시야 밖 오브젝트 제거
---
Occlusion Culling ← 가려진 오브젝트 제거
---
LOD 선택 ← 거리별 메시/텍스처 해상도
---
실제 Draw
```

Meshlet·Mesh Shading은 최신 API(Vulkan 확장 등)에서 강력하지만, **Frustum Culling + LOD + Backface**만 제대로 해도 체감이 크다.

---

## 3. 셰이더·래스터 최적화

픽셀 셰이더는 보통 가장 비싼 단계다. "그리기 전에" 또는 "덜 자주" 실행하는 것이 핵심이다.

| 기법 | 설명 |
|------|------|
| **Early-Z / Depth Pre-pass** | 깊이만 먼저 기록해 뒤쪽 픽셀의 셰이더 실행 스킵 |
| **TBDR** | 모바일 GPU의 타일 단위 지연 렌더링 (온칩 메모리 활용) |
| **Forward+ / Clustered Lighting** | 화면을 클러스터로 나눠 라이트 할당 최적화 |
| **VRS** | 영역별로 프래그먼트 셰이더 실행 빈도 조절 |
| **동적 해상도** | 부하에 따라 렌더 타겟 해상도 조절 |

**정렬 전략**:
- **불투명 오브젝트**: 앞에서 뒤로 (Front-to-back) → Early-Z 효과 극대화
- **반투명 오브젝트**: 뒤에서 앞으로 (Back-to-front) → 블렌딩 정확도

```cmdstack
Depth Pre-pass ← 깊이 버퍼만 기록
---
Color Pass ← 깊이 테스트 통과 픽셀만 셰이딩
---
효과 ← 오버드로우 대폭 감소
```

## 4. 메모리·텍스처 최적화

GPU는 연산보다 **메모리 대역폭**에 막히는 경우가 많다. 텍스처와 메시 데이터가 대역폭의 대부분을 차지한다.

| 기법 | 설명 |
|------|------|
| **텍스처 압축** | BC (데스크톱), ETC2 (모바일), ASTC (범용) |
| **Mipmapping** | 거리에 맞는 mip 레벨 사용 → 캐시 히트율 향상 |
| **텍스처 아틀라스** | 여러 텍스처를 하나로 묶어 바인딩 전환 감소 |
| **메시 최적화** | 정점 캐시·오버드로우·페치 순으로 최적화 |

### meshoptimizer 파이프라인

에셋 임포트 시 아래 순서로 처리하면 대부분의 엔진에서 이득이 난다.

```cmdstack
1. Indexing ← 인덱스 버퍼 생성
2. Vertex Cache Optimization ← 정점 재사용 극대화
3. Overdraw Optimization ← 픽셀 중복 최소화
4. Vertex Fetch Optimization ← 정점 인출 효율
5. Vertex Quantization ← 정점 데이터 양자화
6. Index Filtering ← 불필요 인덱스 제거
7. Shadow Indexing ← 그림자 전용 인덱스 버퍼
```

## 5. 공통 모범 사례

1. **상태 변경 최소화** — 파이프라인·바인딩 전환 횟수를 줄이는 순서로 오브젝트 정렬
2. **큰 배치 우선** — 작은 draw call 여러 개보다 큰 배치 하나
3. **깊이 기반 정렬** — 불투명은 front-to-back, 반투명은 back-to-front
4. **오버드로우 감소** — Early-Z, Depth Pre-pass, VRS
5. **프레임 간 일관성** — 매 프레임 버퍼·텍스처 재할당 금지, lifetime 관리
6. **프로파일링 우선** — 추측으로 최적화하지 말고, 실제 병목을 측정한 뒤 적용

```relflow
CPU 병목
---
Draw call 과다 · 상태 전환 · 커맨드 기록
---
배칭 · 인스턴싱 · GPU-Driven · CB 재사용
---
GPU 병목
---
오버드로우 · 셰이더 비용 · 대역폭
---
컬링 · LOD · Early-Z · 텍스처 압축
```
